# Day 10 — 金融场景工作流模板库

## 今日改动

完成三行业六模板的工作流模板库，含敏感参数加密和参数校验体系，核心工作分五部分：

1. **模板数据结构**：`enterprise/workflows/schemas.py` 定义了 `WorkflowTemplate`（模板名称、适用行业、风险等级、参数定义列表、导航目标描述、预期结果描述、适用审批规则、标签）和 `ParamDefinition`（参数名、类型枚举（6 种）、是否必填、是否敏感、描述、默认值、自定义正则校验）
2. **六个行业模板**：`enterprise/workflows/templates.py` 实现了 6 个覆盖真实高频场景的模板——银行：网银账单自动采集（财务对账/税务申报）和定期贷款还款提醒查询（信贷经理批量跟进）；保险：理赔案件批量状态查询（理赔专员日常作业）和保单到期续保提醒核查（客户经理主动维护）；证券：研报数据自动归档（投研信息整合）和基金净值数据采集（资管产品监控）。全部注册到 `TEMPLATE_REGISTRY` 字典，支持按行业筛选
3. **敏感参数加密**：`enterprise/workflows/crypto.py` 使用 Fernet 对称加密（密钥从 `FINRPA_PARAM_KEY` 环境变量读取），提供 `encrypt_value()` / `decrypt_value()` 进行加解密，`mask_value()` 为 API 响应生成掩码显示（保留首尾字符，中间星号替代）
4. **参数校验引擎**：`enterprise/workflows/validator.py` 实现了三层校验——必填项检查、类型格式校验（URL 正则、日期格式与合法性、整数解析、邮箱格式）、业务规则校验（日期范围不超过 365 天、结束日期晚于开始日期）。支持模板定义中的自定义正则约束
5. **工作流 API**：`enterprise/workflows/routes.py` 提供三个接口——`GET /templates`（列表，可按行业筛选）、`GET /templates/{id}`（详情含完整参数定义）、`POST /instantiate/{id}`（校验参数 → 加密敏感值 → 生成任务 → 返回掩码显示）

## 设计决策

### 模板参数的敏感性标记

每个参数定义携带 `sensitive: bool` 标记，而非在运行时猜测哪些参数是敏感的。原因：

- **确定性**：参数名称不总是能反映敏感性——"account_number" 在银行场景是敏感的，在某些场景可能不是。模板作者在定义参数时明确标记，消除了歧义
- **加密粒度**：只有标记为 sensitive 的参数才会被加密存储，避免对所有参数加密带来的性能开销和调试困难
- **审计协同**：Day 8 的输入值脱敏器可以复用 sensitive 标记来决定脱敏策略，形成一致的数据保护链

### Fernet 对称加密而非非对称加密

选择 Fernet 而非 RSA 等非对称方案：

- **场景适配**：工作流参数的加密和解密都发生在服务端（加密时存储，解密时执行任务），不存在"客户端加密、服务端解密"的需求，对称加密完全满足
- **性能**：Fernet 基于 AES-128-CBC，加解密速度远快于 RSA。RPA 任务可能包含多个敏感参数，逐一加密时性能差异显著
- **密钥管理简洁**：单一密钥通过环境变量注入，支持密钥轮转（重新加密存量数据即可）

### 日期范围上限 365 天

业务规则约束日期范围不超过 365 天（约 12 个月），原因：

- **系统负载**：银行和保险的核心系统在大跨度查询时响应极慢，超过 12 个月的查询可能触发超时或被系统拦截
- **业务合理性**：金融场景下的数据采集通常按月/季/年周期执行，超过 12 个月的单次查询不符合正常操作习惯，可能是误操作
- **可覆盖**：此限制在 `validator.py` 中定义为常量 `MAX_DATE_RANGE_DAYS`，机构可按需调整

### API 响应中的掩码策略

`mask_value()` 保留首尾字符（如 `S**********!`），而非完全掩码（`*************`）：

- **可辨识性**：运维人员可以通过首尾字符确认"参数是否填对了"（如密码首字母是否正确），而无需完整查看密文
- **安全平衡**：仅暴露 2 个字符，对于 8 位以上的密码，暴力搜索空间仅缩小了极小的比例，安全影响可忽略
- **4 字符以下全掩码**：短值（如 PIN 码）全部掩码为 `****`，防止信息泄露

## 金融企业场景下的工程意义

**标准化降低门槛**：金融机构的 IT 部门通常有大量重复性的 RPA 需求——每个支行都需要"账单采集"，每个理赔团队都需要"案件状态查询"。工作流模板将这些高频场景标准化，业务人员选择模板、填写参数即可创建任务，无需 IT 开发介入。

**参数安全的全链路保护**：从用户输入 → Fernet 加密存储 → 任务执行时解密使用 → API 查询时掩码显示 → 审计日志中脱敏记录，敏感参数在每个环节都有对应的保护措施。这形成了完整的"输入 → 存储 → 使用 → 展示 → 审计"安全链路。

**行业模板的实际覆盖率**：6 个模板覆盖了银行/保险/证券三个行业中最高频的查询类和数据采集类场景。这些场景的共同特点是"只读操作为主、风险等级低到中、执行频率高"，是 RPA 自动化的最佳切入点。更高风险的操作（如转账、理赔支付）将在后续模板中逐步覆盖。

## 踩坑记录

1. **Fernet 密钥的编码问题**：`Fernet.generate_key()` 返回 bytes，但环境变量通常存储为 str。`set_key()` 需要同时处理 str 和 bytes 输入，否则测试中直接传入 bytes 会导致 `TypeError`
2. **FastAPI 422 状态码弃用警告**：FastAPI 最新版本将 `HTTP_422_UNPROCESSABLE_ENTITY` 标记为 deprecated，推荐使用 `HTTP_422_UNPROCESSABLE_CONTENT`。当前保持 422 以兼容已有测试，后续统一迁移
